# План реализации Dockform Portable

Этот документ — пошаговый план разработки. Каждый этап — самостоятельная единица работы с явным критерием готовности. Двигаться строго по порядку, не перескакивать. После каждого этапа — ручная проверка по чеклисту перед переходом к следующему.

## Принятые решения

| Решение | Значение |
|---------|----------|
| Каркас приложения | Tauri 2 (Rust + системный WebView2) |
| Фронтенд | React 18 + TypeScript |
| Стилизация | Tailwind CSS + shadcn/ui |
| Word-генерация | docxtemplater (MIT) |
| Автообновление | tauri-plugin-updater + GitHub Releases |
| Хранение шаблонов в облаке | GitHub raw + `manifest.json` с SHA-256 |
| Локальное хранение | Папки рядом с `.exe` |
| Тестирование | Ручное по чеклистам каждого этапа |
| Менеджер пакетов | npm |
| Платформа | Только Windows 10/11 x64 |

## Общий принцип работы по этапам

Каждый этап имеет одну и ту же структуру:
- **Цель** — одно предложение, что мы делаем.
- **Задачи** — конкретный список действий.
- **Критерий готовности** — что должно работать в конце этапа.
- **Антископ** — что в этом этапе НЕ делаем (важно для фокуса).
- **Подсказки** — на что обратить внимание разработчику или Claude Code.

После завершения этапа — отдельный коммит/тег `etap-N-done`.

## Этап 0. Подготовка репозитория

**Цель.** Создать пустой работающий каркас Tauri-приложения.

**Задачи:**
1. Создать новый репозиторий `dockform-portable` на GitHub.
2. Локально: `npm create tauri-app@latest dockform-portable` → выбрать React + TypeScript + npm.
3. Удалить демо-контент из `src/App.tsx`, оставить пустую страницу с заголовком «Dockform».
4. Установить Tailwind: `npm install -D tailwindcss@latest postcss autoprefixer`, `npx tailwindcss init -p`. Настроить `tailwind.config.js`, добавить директивы в `src/index.css`.
5. Установить shadcn/ui: `npx shadcn@latest init` (выбрать Tailwind, путь компонентов `@/components/ui`).
6. Установить начальный набор компонентов: `npx shadcn@latest add button card input label radio-group select textarea dialog alert`.
7. Создать структуру папок:
   ```
   /
   ├── src/                  # фронтенд
   │   ├── components/
   │   ├── lib/
   │   └── types/
   ├── src-tauri/            # Rust-бэкенд
   ├── templates/
   │   ├── pretenzii/
   │   └── documentation/
   ├── dictionaries/
   ├── docs/
   ├── scripts/
   ├── CLAUDE.md
   ├── ARCHITECTURE.md
   ├── plan.md
   ├── README.md
   ├── manifest.json
   └── killswitch.json
   ```
8. Положить пустые `manifest.json` (`{"templates": {}, "dictionaries": {}}`) и `killswitch.json` (`{"active": true, "message": ""}`).
9. Настроить `.gitignore` (target/, node_modules/, dist/, cache/, output/, *.lock).
10. Проверить, что `npm run tauri dev` открывает окно приложения с заголовком «Dockform».
11. Первый коммит: `git commit -m "Этап 0: каркас Tauri + React + Tailwind + shadcn"`.

**Критерий готовности.** `npm run tauri dev` открывает пустое окно «Dockform» без ошибок в консоли. Структура папок создана.

**Антископ.** Ничего, что связано с шаблонами, формами, обновлениями — всё это в следующих этапах.

**Подсказки.** На этом этапе принципиально работать в `npm run tauri dev` (не в обычном `npm run dev`), чтобы убедиться, что десктоп-окно собирается. Если возникнут проблемы с WebView2 — установить отдельно с сайта Microsoft.

## Этап 1. Скелет UI и навигация

**Цель.** Реализовать каркас навигации: режимы, ярлыки, экраны выбора.

**Задачи:**
1. В `src-tauri/src/main.rs` научиться читать аргументы командной строки и передавать `mode` во фронтенд через Tauri command.
2. Во фронтенде создать стейт `mode: 'pretenzii' | 'documentation' | null`.
3. Если `mode === null` (запуск без аргумента) — показать экран выбора режима с двумя большими карточками.
4. Если `mode` задан — показать экран со списком шаблонов (пока заглушка-массив прямо в коде):
   ```ts
   const fakeTemplates = [
     { id: 't1', title: 'Шаблон 1', description: 'Описание' },
     { id: 't2', title: 'Шаблон 2', description: 'Описание' },
   ];
   ```
5. Заголовок окна меняется в зависимости от режима: «Dockform — Претензионная работа» / «Dockform — Документация» / «Dockform».
6. Кликом по карточке шаблона переходим на экран-заглушку «Здесь будет форма для {название}».
7. Реализовать запоминание размера окна в `cache/window.json` (создать в Rust). Восстанавливать при запуске.
8. Заглушка для иконки приложения: разные иконки для двух режимов (можно из заглушки lucide-react `Briefcase` и `FileText`).

**Критерий готовности.**
- Запуск `dockform.exe --mode=pretenzii` сразу открывает список шаблонов в нужном разделе.
- Запуск `dockform.exe` без аргументов показывает экран выбора режима, после клика — список шаблонов.
- Заголовок окна корректный.
- При закрытии окна с изменённым размером и повторном открытии — размер сохраняется.

**Антископ.** Никакого чтения файлов с диска, никаких реальных шаблонов. Всё на mock-данных.

**Подсказки.** Для аргументов командной строки в Tauri 2 удобно использовать `tauri::AppHandle::env().args` или передавать через `--mode=...` и парсить руками. Состояние режима не нужно хранить в файле — он определяется только аргументом запуска.

## Этап 2. Чтение шаблонов с диска

**Цель.** Заменить заглушки на реальное чтение `templates/{mode}/` и парсинг YAML-конфигов.

**Задачи:**
1. Установить `serde`, `serde_yaml` в Rust, `js-yaml` или `yaml` во фронтенд (логику парсинга лучше делать на Rust, отдавать на фронт уже типизированные структуры).
2. Определить TypeScript-типы и Rust-структуры для конфига шаблона:
   ```ts
   type FieldType = 'text' | 'textarea' | 'number' | 'radio' | 'dropdown' | 'dictionary' | 'date';
   type FieldConfig = { name: string; label: string; type: FieldType; required?: boolean; options?: string[]; visible_if?: string; ... };
   type TemplateConfig = { template: string; title: string; description?: string; output_filename: {...}; fields: FieldConfig[]; };
   ```
3. В Rust написать команду `list_templates(mode: String) -> Vec<TemplateConfig>`:
   - Сканирует `templates/{mode}/` рядом с `.exe`.
   - Для каждого `*.yaml` парсит конфиг, проверяет, что соответствующий `.docx` существует.
   - Возвращает массив. Невалидные конфиги — в лог, не падать.
4. Во фронтенде вызвать `list_templates` при входе в раздел, отрисовать список карточек.
5. Если папка пуста или нет валидных шаблонов — показать пустое состояние «Шаблоны не найдены. Проверьте, что папка templates/{mode}/ существует и содержит .docx + .yaml файлы.»
6. Положить в `templates/documentation/` один тестовый шаблон `test.docx` (с одним полем `{{ имя }}`) и `test.yaml` для проверки:
   ```yaml
   template: test.docx
   title: "Тестовый шаблон"
   description: "Для проверки этапа 2"
   output_filename:
     pattern: "test_{date}.docx"
     fields: []
   fields:
     - name: имя
       label: "Имя"
       type: text
       required: true
   ```

**Критерий готовности.** Положенный в `templates/documentation/` `.docx` + `.yaml` появляются в списке. Удаление файлов из папки убирает их из списка после перезапуска. Невалидный YAML не роняет приложение.

**Антископ.** Форма не рендерится, шаблон только в списке.

**Подсказки.** Путь до `.exe` в Tauri: `tauri::api::path::resource_dir` не подходит для портабла, нужно `std::env::current_exe()` → parent. Все пути — относительно этой папки.

## Этап 3. Динамический рендер формы

**Цель.** По YAML-описанию шаблона построить рабочую форму с валидацией.

**Задачи:**
1. Подключить `react-hook-form` + `zod` для валидации.
2. Создать компонент `<DynamicForm config={templateConfig} onSubmit={...} />`.
3. Реализовать рендер для типов полей: `text`, `textarea`, `number`, `radio`, `dropdown`, `date` (использовать соответствующие shadcn-компоненты).
4. Реализовать `visible_if` — простой DSL вида `"поле == 'значение'"` или `"поле != 'значение'"`. Парсер минимальный, без сложных выражений. Поле скрыто → не отправляется в результат.
5. Валидация: обязательные поля (`required: true`), числа (`type: number`), даты.
6. Кнопка «Сформировать документ» внизу. По клику — собрать значения и пока просто `console.log` результат.
7. Кнопка «Назад к списку шаблонов» сверху.
8. Расширить тестовый YAML на все типы полей и проверить.

**Критерий готовности.** Форма рендерится, валидация работает, скрытие полей по `visible_if` работает, по сабмиту в консоль выводится JSON со значениями.

**Антископ.** Тип поля `dictionary` пока не реализуем — на этапе 5. Нет сохранения черновика — этап 5. Нет генерации документа — этап 4.

**Подсказки.** `visible_if` парсить нет смысла регулярками — проще сделать функцию `evaluate(expr: string, formData: Record<string, any>): boolean`, которая разбирает три случая: `==`, `!=`, `truthy` (просто имя поля). Этого хватит на все юридические сценарии.

## Этап 4. Генерация документа

**Цель.** Из заполненной формы получить реальный `.docx` на диске.

**Задачи:**
1. Установить `docxtemplater` и `pizzip`: `npm install docxtemplater pizzip`.
2. Создать модуль `src/lib/generator.ts` с функцией `generateDocx(templatePath: string, data: Record<string, any>): Promise<Uint8Array>`.
3. Чтение `.docx` с диска — через Tauri command `read_template(path: string) -> Vec<u8>` (Rust → фронт), потому что docxtemplater работает в JS.
4. Сборка имени файла по `output_filename.pattern`: подставить значения полей, заменить `{date}` на текущую дату в формате `ГГГГ-ММ-ДД`. Если поле из `output_filename.fields` пустое — показать ошибку «Заполните поля для имени файла» с пометкой нужных полей.
5. Открыть системный диалог «Сохранить как…» через `@tauri-apps/api/dialog` `save()` с предзаполненным именем.
6. Записать файл в выбранный путь через Tauri command `write_file(path: String, bytes: Vec<u8>)`.
7. После успешной записи — модальное окно «Документ сохранён» с двумя кнопками: «Открыть папку» (вызывает Rust команду `open_in_explorer(path)` через `std::process::Command::new("explorer.exe").arg(path)`) и «ОК».
8. Положить в тестовый `.docx` простую структуру с парой переменных и проверить полный цикл.

**Критерий готовности.** Заполнил форму → нажал «Сформировать» → выбрал папку → файл реально создан, открывается в Word, переменные подставлены.

**Антископ.** Никакой логики условных блоков `{#field}...{/field}` пока не используем в шаблонах (это работает в docxtemplater из коробки, но пока тестируем простую подстановку).

**Подсказки.** docxtemplater по умолчанию использует разделители `{` и `}`. Для всего проекта выбрать **одинарные `{`** — это стандарт docxtemplater, меньше шансов на ошибку с условными блоками. Простая подстановка тогда выглядит как `{нмцк_цифра}`, условный блок как `{#применяется_запрет}текст{/применяется_запрет}`. При миграции старых шаблонов из docxtpl (где было `{{ переменная }}`) — заменить на одинарные.

## Этап 5. Справочники, dictionary и черновик формы

**Цель.** Реализовать автозаполнение из справочников и сохранение черновика.

**Задачи:**

### 5.1. Справочники
1. В Rust-команде `list_templates` дополнительно загружать все JSON из `dictionaries/` в общий стор.
2. Реализовать тип поля `dictionary`:
   ```yaml
   - name: zam
     label: "Заместитель"
     type: dictionary
     source: deputies
     display: "{fio} ({position})"
     fills:
       Фам_Имя_Отч_зама_им_пад: fio_im
       Фам_Имя_Отч_зама_род_пад: fio_rod
       должность_зама: position
   ```
3. Рендерится как `<Combobox>` (shadcn) со списком из соответствующего справочника. `display` — паттерн отображения опции.
4. При выборе опции — автоматически заполнить поля, перечисленные в `fills`.
5. Поля, заполненные через `fills`, всё равно редактируемы (на случай, если в справочнике опечатка и надо поправить разово).
6. Создать тестовый `dictionaries/deputies.json` с 2–3 записями.

### 5.2. Черновик
1. Стор черновика: `cache/drafts/{mode}__{template_filename_without_ext}.json`.
2. При любом изменении значения поля — `setTimeout` на 500 мс, потом запись в файл (Tauri command `save_draft(key: String, json: String)`).
3. При открытии формы шаблона — проверить наличие соответствующего черновика. Если есть и непустой — показать тост вверху формы:
   ```
   Найден незавершённый черновик от ДД.ММ.ГГГГ ЧЧ:ММ.
   [Восстановить] [Начать заново]
   ```
4. «Восстановить» — заполнить форму значениями из черновика. Перед заполнением сверить набор полей: если в шаблоне теперь есть/нет каких-то полей по сравнению с черновиком — показать предупреждение «Шаблон обновился, часть полей не подставится».
5. «Начать заново» — удалить файл черновика.
6. После успешной генерации документа в Этапе 4 — удалять черновик автоматически.

**Критерий готовности.** Выбор зама автозаполняет три поля. Закрыл приложение посреди заполнения, открыл снова, увидел тост, восстановил, доделал, сгенерировал — черновик удалился.

**Антископ.** Никакого облачного синка черновиков. Только локально.

**Подсказки.** Для записи в `cache/drafts/` нужно создать папку, если её нет. Делать это в Rust один раз при старте приложения.

## Этап 6. Сетевые функции (Kill Switch + обновления)

**Цель.** Реализовать удалённую блокировку, обновление шаблонов и обновление приложения.

**Задачи:**

### 6.1. Kill Switch
1. На старте приложения в Rust сделать запрос на `https://raw.githubusercontent.com/{user}/{repo}/main/killswitch.json` с таймаутом 3 сек.
2. Если ответ получен и `active === false` — заморозить UI, показать модальное окно с текстом из `message` и единственной кнопкой «Закрыть».
3. Если интернета нет / запрос провалился — продолжить работу в обычном режиме (последнее известное состояние). Кэшировать предыдущий ответ в `cache/last_killswitch.json` и использовать его, если новый не получен.
4. URL репозитория зафиксировать в `tauri.conf.json` или в отдельном `app-config.json` рядом с `.exe` (последнее даёт больше гибкости — можно поправить без пересборки).

### 6.2. Обновление шаблонов
1. На старте после kill switch — скачать `manifest.json` с того же репо.
2. Сравнить с локальным `cache/last_manifest.json`. Для каждого изменившегося/нового файла — скачать через raw-ссылку и положить в `templates/` или `dictionaries/`.
3. Удалённые из манифеста файлы — удалить локально.
4. Если в этот момент пользователь уже находится в форме (а не в списке шаблонов) — отложить обновление до возвращения в список ИЛИ показать ненавязчивый тост в правом нижнем углу «Шаблон обновился, перезагрузить форму?».
5. Сохранить новый `manifest.json` как `cache/last_manifest.json`.
6. Написать вспомогательный Python-скрипт `scripts/build_manifest.py`: проходит по `templates/` и `dictionaries/`, пересчитывает SHA-256, обновляет `manifest.json` в корне репо. Запускается перед каждым коммитом шаблонов.

### 6.3. Обновление приложения
1. Установить `tauri-plugin-updater`: добавить в зависимости Rust, `npm install @tauri-apps/plugin-updater` во фронтенд.
2. Сгенерировать пару ключей `tauri signer generate`. Приватный — в надёжное место (НЕ в репо), публичный — в `tauri.conf.json`.
3. Настроить `tauri.conf.json` секция `updater`:
   ```json
   {
     "createUpdaterArtifacts": true,
     "endpoints": ["https://github.com/{user}/{repo}/releases/latest/download/latest.json"],
     "pubkey": "..."
   }
   ```
4. На старте после обновления шаблонов — `check()` плагина updater. Если есть новая версия — спросить пользователя «Доступно обновление приложения. Установить сейчас?». При согласии — `downloadAndInstall()` с прогрессом, потом перезапуск.
5. Документировать процесс выпуска релиза в `docs/release.md` (создать в Этапе 7): bump version → `tauri build` → подписать → загрузить артефакты на GitHub Releases с тегом версии → сгенерировать `latest.json`.

**Критерий готовности.**
- Установка `"active": false` в `killswitch.json` репо → следующий запуск приложения у тестового пользователя блокируется.
- Изменение шаблона в репо + пересчёт `manifest.json` + push → следующий запуск подтягивает новый шаблон.
- Выпуск нового релиза приложения → следующий запуск предлагает обновиться.

**Антископ.** Отдельных «бета»-каналов и сложного rollout нет. Один main, одна версия для всех.

**Подсказки.** Все сетевые запросы — с таймаутом и обработкой ошибок. Приложение должно стартовать и без интернета максимум за 3–5 секунд (не 30).

## Этап 7. Калькуляторы и производные значения в формах

**Цель.** Дать шаблонам возможность подставлять в документ значения, автоматически выводимые из полей формы: расчёты (неустойка, проценты по ст. 395 ГК, пени) и форматные производные (например, сумма прописью к числовому полю). Универсальные калькуляторы переиспользуются между шаблонами, специфичные — пишутся под один шаблон. Все объявления — **внутри самого YAML-шаблона**, в обычном списке `fields:` (без отдельной верхней секции).

**Задачи:**

### 7.1. Тип поля `calculator` и расширение YAML
1. Добавить `"calculator"` в `FieldType` (`src/types/index.ts`).
2. Расширить `FieldConfig` тремя опциональными полями:
   ```ts
   calculator?: string;                   // имя калькулятора в реестре
   inputs?: Record<string, string>;       // вход_калькулятора → имя_поля_формы
   outputs?: Record<string, string>;      // выход_калькулятора → имя_переменной_для_docx
   ```
3. Пример записи в YAML-шаблоне:
   ```yaml
   - name: неустойка_блок
     label: "Расчёт неустойки"
     type: calculator
     calculator: penalty_44fz_part6
     inputs:
       сумма_контракта: сумма_контракта
       дней_просрочки: дней_просрочки
       ставка_цб: ставка_цб
     outputs:
       сумма_неустойки: сумма_неустойки
       расчёт_подробно: расчёт_неустойки_текст
     visible_if: "тип_нарушения == 'просрочка'"
   ```

### 7.2. Реестр калькуляторов
1. Создать `src/lib/calculators/` с файлами:
   - `types.ts` — `type Calculator = (inputs: Record<string, unknown>) => Record<string, unknown>` плюс тип-описание (id, ожидаемые входы и выходы — для валидации YAML на старте).
   - `index.ts` — `export const CALCULATORS: Record<string, Calculator>` и хелпер `runCalculator(name, inputs)`.
2. Универсальные калькуляторы заводятся по факту, когда понадобятся первому шаблону. Стартовый набор для претензий:
   - `penalty-44fz-part6.ts` — неустойка по 44-ФЗ ч.6 ст.34 (ставка ЦБ × дней просрочки / 300).
   - `penalty-223fz.ts` — болванка под 223-ФЗ.
   - `interest-395-gk.ts` — проценты по ст. 395 ГК РФ.
3. Специфичные калькуляторы — рядом, по соглашению об именовании `penalty-<id_шаблона>.ts`. Для системы они от универсальных ничем не отличаются — просто другой ключ в том же реестре.
4. Каждый калькулятор покрыть мини-тестом на 2–3 кейса (граничные значения, ноль входов). Тест-раннер — vitest (легче ставится в Tauri-проект, чем jest).

### 7.3. Интеграция в `DynamicForm`
1. Ветка рендера для `type: calculator`: read-only карточка с подписями выходов и их текущими значениями.
2. Пересчёт через `useWatch` react-hook-form по полям из `inputs`. Дебаунс 150–300 мс, чтобы не дёргать функцию на каждый символ.
3. Если каких-то входов ещё нет / поля пустые — показать заглушку «Заполните поля выше», не падать.
4. Выходы калькулятора писать в значения формы через `setValue(имя_переменной_для_docx, …)` — так они автоматически попадут в `src/lib/generator.ts` при подстановке в `.docx` без правок генератора.
5. `visible_if` уважать как у обычных полей.

### 7.4. Валидация и ошибки
1. При загрузке шаблона — проверить, что для каждого `type: calculator` указан существующий `calculator: <name>` и что объявленные `inputs`/`outputs` совпадают с метаданными калькулятора. Несовпадение — явная ошибка с указанием файла шаблона.
2. Если калькулятор бросил исключение (плохие входные данные) — превью показывает ошибку, переменные `outputs` остаются пустыми, форма не блокируется.

### 7.5. Суммы прописью (формат-производная от числового поля)
Особый случай производной: одно число → одна строка прописью. Не делаем через калькулятор (overhead с маппингом `inputs`/`outputs` не нужен — трансформация одна и та же для любых сумм), а вводим простой атрибут у поля `number`.

1. Расширить `FieldConfig`:
   ```ts
   text_output?: string;   // имя парной переменной для прописи; пишется автоматически
   ```
   Применимо к `type: number` (в первую очередь под суммы в рублях). Пример YAML:
   ```yaml
   - name: сумма_контракта
     label: "Сумма контракта, руб."
     type: number
     text_output: сумма_контракта_прописью
   ```
   В `.docx`: `{сумма_контракта} ({сумма_контракта_прописью}) руб.` → `3451,00 (три тысячи четыреста пятьдесят один рубль 00 копеек) руб.`
2. Реализовать функцию `amountToWords(num: number): string` в `src/lib/amount-to-words.ts`. Возвращает форму «N рублей K копеек» с правильными склонениями (один рубль / два рубля / пять рублей). Сначала пробуем готовый npm-пакет (например, `rubles` или аналог) — если не подходит по точности склонений / размеру, пишем своими ~80 строк. Зависимость согласовать перед установкой (правило №7 из CLAUDE.md).
3. В `DynamicForm` для поля с `text_output` — подписаться на `useWatch` по имени поля и через `setValue(text_output, amountToWords(value))` обновлять парную переменную. Под полем — read-only превью текущей прописи (чтобы пользователь видел результат и мог отловить ошибку ввода).
4. На загрузке шаблона валидировать: `text_output` допустим только для `type: number`; имя переменной не должно конфликтовать с другими полями.
5. Юнит-тесты для `amountToWords`: 0, 1, 2, 5, 21, 100, 1000, 1234567, дробные копейки, отрицательные (отклонять или показывать «минус» — выбрать на месте, зафиксировать в тесте).

### 7.6. Документация
1. Дополнить `docs/template-authoring.md` разделами «Калькуляторы» (универсальные/специфичные, входы/выходы) и «Суммы прописью» (как пометить поле `text_output` и какую переменную вставлять в `.docx`).
2. Дополнить `CLAUDE.md` секцию «Где что лежит» пунктами:
   - «Новый калькулятор → `src/lib/calculators/<name>.ts` + регистрация в `index.ts`».
   - «Сумма прописью у нового шаблона → ставим `text_output: <имя_прописью>` у `type: number` в YAML, никакого кода не нужно».

**Критерий готовности.**
- Минимум один шаблон в `templates/pretenzii/` использует калькулятор неустойки, и сгенерированный `.docx` содержит правильно посчитанную сумму и текст расчёта.
- Тот же калькулятор используется ещё в одном шаблоне без изменений — доказательство универсальности.
- Изменение поля-входа в форме сразу обновляет превью.
- Удаление калькулятора из реестра при загрузке шаблона даёт понятную ошибку, а не молчаливый сбой.
- В каком-либо шаблоне есть числовое поле с `text_output`, в готовом `.docx` пропись подставляется корректно (склонения «рубль/рубля/рублей» и «копейка/копейки/копеек» правильные).

**Антископ.** Не пишем универсальный движок выражений в YAML (типа `formula: "сумма * дни / 300"`). Логика — в TS-коде калькулятора. Не вводим UI-конструктор калькуляторов. Калькулятор не изменяет введённые пользователем данные — только производит новые значения для подстановки в документ.

**Подсказки.** Не плодить калькуляторы под каждый микро-случай. Если разница между шаблонами — только в одной константе или коэффициенте, проще сделать один универсальный калькулятор с дополнительным числовым входом, чем два почти одинаковых. Специфичный калькулятор оправдан, когда формула реально не повторяется нигде.

## Этап 8. Упаковка, миграция и финализация

**Цель.** Получить готовый портативный `.exe`, перенести реальные шаблоны, передать в эксплуатацию.

**Задачи:**

### 8.0. Ключи подписи и первая проверка автообновления
Обвязка обновления приложения сделана на этапе 6.3, но до первого подписанного релиза нет реальной пары ключей и нет артефактов на GitHub Releases. Здесь это закрываем.

1. Сгенерировать пару ключей: `npm run tauri signer generate -- -w ~/.tauri/dockform.key`. Пароль — обязательно (не пустой); сохранить отдельно (1Password / бумажка). Приватный ключ остаётся в `~/.tauri/`, в репозиторий НЕ попадает.
2. Скопировать содержимое `~/.tauri/dockform.key.pub` в `src-tauri/tauri.conf.json` → `plugins.updater.pubkey` (вместо текущего `REPLACE_BEFORE_FIRST_RELEASE`).
3. Перед `tauri build` выставить переменные окружения с приватным ключом и паролем:
   ```bash
   export TAURI_SIGNING_PRIVATE_KEY="$(cat ~/.tauri/dockform.key)"
   export TAURI_SIGNING_PRIVATE_KEY_PASSWORD="..."
   ```
4. Собрать первый релиз, выложить артефакты + `latest.json` на GitHub Releases с тегом версии (детали — в 8.3 `docs/release.md`).
5. Проверить критерий из этапа 6: bump version → новый релиз → следующий запуск приложения у тестового пользователя предлагает обновиться.

### 8.1. Сборка портабла
1. В `tauri.conf.json` для портабл-сборки настроить `bundle.windows.webviewInstallMode`. Для портабла оптимально `"downloadBootstrapper"` (если WebView2 нет — скачается при первом запуске) или `"skip"` (если WebView2 точно есть у всех).
2. Альтернатива: собрать через `tauri build`, взять файл из `src-tauri/target/release/dockform.exe`, упаковать вручную в zip с папками `templates/`, `dictionaries/` и пустыми `cache/`, `output/`.
3. Создать два `.lnk` ярлыка с аргументами `--mode=pretenzii` и `--mode=documentation` через PowerShell-скрипт `scripts/make_shortcuts.ps1`.
4. Финальная папка распространения:
   ```
   dockform/
   ├── dockform.exe
   ├── Претензионная.lnk
   ├── Документация.lnk
   ├── templates/...
   ├── dictionaries/...
   ├── cache/
   ├── output/
   └── README.txt              # короткая инструкция для конечного пользователя
   ```

### 8.2. Миграция шаблонов из старого проекта (полу-автомат)
1. Скопировать из старого `dockform/templates/*.docx` в новый `templates/documentation/`.
2. Скопировать YAML-конфиги, привести к новому формату (добавить `output_filename`, тип `dictionary` для замов, `visible_if` где надо).
3. Написать скрипт `scripts/migrate_from_n8n.py`:
   - Принимает на вход путь к старому `n8n-workflow.json`.
   - Парсит ноды `Edit Fields1` и `Edit Fields4`, извлекает все условные тексты.
   - Выдаёт `migration_report.md`: для каждого «сборного» поля (`Запрет1875`, `Ограничение1875`, `Преференции1875`, `Уголовники_текст`, `Инвалиды`, `Эквивалент`) — какое исходное Yes/No-поле им управляет, и какие тексты идут в каждом случае.
4. Написать скрипт `scripts/replace_placeholders.py`:
   - Принимает: путь к `.docx`, имя плейсхолдера (например, `Запрет1875`), имя управляющего поля (`применяется_запрет`), текст для «Да», текст для «Нет».
   - Через `python-docx` находит плейсхолдер в документе и заменяет его на блок:
     ```
     {#применяется_запрет}текст для Да{/применяется_запрет}{^применяется_запрет}текст для Нет{/применяется_запрет}
     ```
   - Если плейсхолдер разбит на несколько runs из-за форматирования — выдать предупреждение и пропустить, потребовать ручной правки.
5. Прогнать все шаблоны через скрипт. Открыть каждый в Word, проверить визуально. Доделать руками то, что скрипт пропустил.
6. Тестирование: заполнить каждый шаблон тестовыми данными в собранном `.exe`, сверить результат с эталонным документом из старого проекта.

### 8.3. Документация для будущего себя
1. Написать `docs/template-authoring.md` — как добавлять новый шаблон с нуля (см. отдельный документ).
2. Написать `docs/release.md` — как выпускать новый релиз (bump → build → sign → upload).
3. Написать `README.txt` для папки портабла (1 страница, для конечного пользователя): «Запускайте ярлык. Если не работает — попробуйте установить WebView2 с сайта Microsoft».

### 8.4. Передача в эксплуатацию
1. Положить готовую папку на общий ресурс / отдать через флешку / архив 5–10 сотрудникам.
2. Прогнать с одним пилотным пользователем полный цикл (запуск → выбор шаблона → заполнение → получение `.docx`).
3. Зафиксировать обнаруженные проблемы, поправить, выпустить v1.0.0.

**Критерий готовности.** Сотрудник вставляет флешку, кликает «Документация.lnk», заполняет форму, получает реальный документ, открывает его в Word — всё корректно. Полный цикл работает офлайн.

**Антископ.** Многоплатформенная сборка, App Store, MSIX. Только portable .exe для Windows.

**Подсказки.** На этом этапе главное — не пытаться сделать миграцию идеально-автоматической. 80% автоматизации + 20% ручной работы быстрее, чем 100% автоматизации с написанием парсера для всех краевых случаев Word XML.
