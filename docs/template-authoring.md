# Добавление нового шаблона

Эта инструкция описывает полный цикл подготовки нового Word-шаблона: от чистого `.docx` до коммита в репозиторий, после которого шаблон автоматически прилетит ко всем пользователям. Делается это редко (~раз в полгода), поэтому проще не вспоминать каждый раз, а идти по этому документу шаг за шагом.

## Когда нужен новый шаблон

- Появился новый тип документа (например, новый вид претензии).
- Существенно изменился старый — настолько, что проще создать новую версию, чем редактировать.
- При простой правке текста существующего шаблона новый файл не нужен — просто редактируем имеющийся `.docx`, обновляем манифест, коммитим (см. секцию 6).

## Шаг 1. Подготовка Word-файла

### 1.1. Создание документа

1. Открыть Word, написать содержание шаблона (или вставить из имеющегося документа).
2. Сохранить как `.docx` (не `.doc`) в формате Word Document. Имя файла — латиницей или транслитом, snake_case, без пробелов: `pretenziya_po_srokam.docx`, `44_uslugi_smp.docx`.
3. Положить пока в любую временную папку — в репозиторий копируем в самом конце, после валидации.

### 1.2. Расстановка плейсхолдеров

В местах, где должны подставляться данные из формы, пишем плейсхолдеры в фигурных скобках:

```
ИКЗ закупки: {ikz}
Начальная максимальная цена контракта: {нмцк_цифра} ({нмцк_текст}) рублей.
Заместитель: {должность_зама} {Фам_Имя_Отч_зама_им_пад}.
```

**Имена переменных могут быть на русском.** Это унаследовано из старого проекта и поддерживается docxtemplater. Латиница тоже допустима. Что важно:

- Без пробелов в имени: `{нмцк_цифра}`, не `{нмцк цифра}`.
- Без точек, дефисов и спецсимволов. Допустимы буквы, цифры, подчёркивание.
- Регистр имеет значение: `{ИКЗ}` и `{ikz}` — разные переменные.
- **Разделители — одинарные `{` и `}`.** НЕ `{{ }}` как было в старом проекте на docxtpl.

### 1.3. Условные блоки

Когда часть текста должна появляться только при определённом ответе в форме — оборачиваем её в условный блок:

```
{#применяется_запрет}
Запрет на допуск товаров иностранного производства устанавливается согласно п. {пункт_тз_запрет} ТЗ.
{/применяется_запрет}

{^применяется_запрет}
Запрет на допуск товаров не применяется.
{/применяется_запрет}
```

- `{#field}...{/field}` — блок показывается, если значение поля «истинно» (любая непустая строка кроме `'Нет'`, `'false'`, `0`).
- `{^field}...{/field}` — инверсный, показывается если значение «ложно».
- Открывающий и закрывающий теги должны быть в одном и том же абзаце или на одном уровне вложенности. Если внутри блока есть несколько абзацев — открывающий тег на первом, закрывающий на последнем.

### 1.4. Подводные камни Word

**Проблема: плейсхолдер разбит на runs.** Когда юрист (или вы сами) редактирует документ в Word, в XML-структуре `.docx` плейсхолдер `{переменная}` может оказаться разбитым на несколько «runs» из-за изменений форматирования (например, часть слова была выделена курсивом и потом возвращена обратно). Глазами этого не видно, но docxtemplater не может найти такой плейсхолдер и выдаёт ошибку.

**Как избежать:**
- Печатайте плейсхолдер сразу полностью одной командой, без правок внутри него.
- Если плейсхолдер скопирован из другого документа — выделите его и нажмите `Ctrl+Space` (сброс форматирования к стилю абзаца). Это объединит runs.
- Можно весь шаблон выделить целиком (`Ctrl+A`) и применить `Ctrl+Space`. Но осторожно — это сбросит все ручные выделения курсивом, жирным и т.п.

**Как обнаружить:** см. шаг 3 ниже про `extract_variables.py`.

### 1.5. Быстрая разметка через автозамену (`scripts/replace_placeholders.py`)

Расставлять `{плейсхолдеры}` и условные блоки `{#}`/`{^}` вручную в Word муторно и
легко разбить на runs. Альтернатива: разметить образец простой нотацией и прогнать
скрипт — он сам проставит синтаксис docxtemplater. Так готовился блок
`templates/pretenzii/44_аук_единовр_пост_*`.

**Как размечать образец:**

1. **Значение** оборачиваем в угловые скобки — пример прямо в тексте:
   ```
   <30.03.2026>   <ООО «Поставщик»>   <N 90 ОАЭФ/… от 07.11.2025>
   ```
   Скрипт заменит i-й такой `<…>` на i-й плейсхолдер из переданного списка.
   (Порядок важен — поэтому удобнее сперва `--check`, см. ниже.)

2. **Условный / опциональный текст** — маркерами `<если:поле>…<иначе>…<конец>`:
   ```
   <если:обеспечение_нг>… по независимой гарантии …<иначе>… из обеспечения …<конец>
   <если:почтовый_адрес>Почтовый адрес: <почтовый_адрес><конец>
   ```
   → `{#поле}…{/поле}{^поле}…{/поле}` и `{#поле}…{/поле}`. `<иначе>` необязателен.
   Эти маркеры не занимают позицию в списке значений.
   Помни про конвенцию truthy: ответ `«Нет»` у `radio` приводится к пустому
   (блок `{#поле}` скрыт, `{^поле}` показан) — см. CLAUDE.md и `formatValuesForDocx`.

3. **Примечания себе** — родными комментариями Word (Рецензирование → Создать
   примечание). Они в отдельном файле, не печатаются и скриптом не трогаются.
   `@…@` прямо в тексте больше не используем.

**Как прогнать:**

```bash
# 1) посмотреть, какие спаны-значения нашлись и в каком порядке
python3 scripts/replace_placeholders.py --check templates/.../образец.docx
```

Дальше — программно (список плейсхолдеров по порядку спанов; условные маркеры
скрипт обработает сам):

```python
import sys; sys.path.insert(0, "scripts")
import replace_placeholders as rp
rp.process("templates/.../образец.docx",
           ["{наименование_контрагента}", "{дата_поставки}", ...],
           out_path="templates/.../итог.docx")
```

Один спан-значение может содержать несколько плейсхолдеров
(`"№ {номер_договора} от {дата_договора}"`) — это нормально.

**Плюсы:** маркер можно печатать как угодно (хоть `<имя>` целиком, хоть с
форматированием — скрипт устойчив к дроблению на runs), `Ctrl+Space` не обязателен.
Всё равно полезно проверить результат рендером (см. шаг 3.3).

## Шаг 2. Написание YAML-конфига

Рядом с `.docx` создаётся файл с таким же именем, но расширением `.yaml`:
```
pretenziya_po_srokam.docx
pretenziya_po_srokam.yaml
```

### 2.1. Базовая структура

```yaml
template: pretenziya_po_srokam.docx     # имя .docx в этой же папке
title: "Претензия по нарушению сроков"  # отображается в списке шаблонов
description: "Контрагент нарушил срок поставки/выполнения работ"  # подзаголовок карточки

output_filename:
  pattern: "{договор}_{контрагент}_{date}.docx"  # шаблон имени файла при сохранении
  fields: [договор, контрагент]                   # обязательные поля для генерации имени

fields:
  - name: договор
    label: "Номер договора"
    type: text
    required: true

  # ... остальные поля
```

`{date}` в `pattern` подставляется автоматически в формате `ГГГГ-ММ-ДД`.

### 2.2. Типы полей

**`text`** — однострочный ввод.
```yaml
- name: ikz
  label: "ИКЗ"
  type: text
  required: true
  placeholder: "26 цифр"
  help_text: "Идентификационный код закупки"
```

**`textarea`** — многострочный текст (для длинных описаний предмета закупки, оснований и т.п.).
```yaml
- name: предмет_закупки
  label: "Предмет закупки"
  type: textarea
  required: true
```

**`number`** — число. Валидация на числовой формат.
```yaml
- name: нмцк_цифра
  label: "НМЦК (цифрой)"
  type: number
  required: true
```

**`radio`** — выбор одного из 2-4 вариантов (отображается как радиокнопки в строку).
```yaml
- name: применяется_запрет
  label: "Применяется запрет по 1875-ФЗ?"
  type: radio
  options: ["Да", "Нет"]
  required: true
```

**`dropdown`** — выбор одного из многих вариантов (отображается как раскрывающийся список).
```yaml
- name: год_закупки
  label: "Год закупки"
  type: dropdown
  options: ["2025", "2026", "2027"]
  required: true
```

**`date`** — выбор даты через календарь.
```yaml
- name: дата_претензии
  label: "Дата претензии"
  type: date
  required: true
```

**`dictionary`** — выбор из справочника с автозаполнением связанных полей. См. подробнее в 2.4.

**`calculator`** — read-only поле, которое не редактируется пользователем, а считает значение из других полей и кладёт результат в одну или несколько переменных для шаблона. См. подробнее в 2.6.

### 2.3. Условная видимость (`visible_if`)

Поле появляется только при выполнении условия. Поддерживаются операторы `==` и `!=`, значение в одиночных кавычках:

```yaml
- name: применяется_запрет
  label: "Применяется запрет?"
  type: radio
  options: ["Да", "Нет"]

- name: пункт_тз_запрет
  label: "Пункт ТЗ для запрета"
  type: text
  visible_if: "применяется_запрет == 'Да'"
```

Скрытое поле не сохраняется в форму и не передаётся в шаблон.

### 2.4. Справочники (`dictionary` + `fills`)

Когда выбор одного значения должен автоматически заполнить несколько полей в шаблоне (например, выбор зама → подставить его ФИО в им. и род. падежах + должность), используем тип `dictionary`.

**Шаг А.** Завести/обновить справочник в `dictionaries/{name}.json`:
```json
[
  {
    "fio": "Иванов И.И.",
    "fio_im": "Иванов Иван Иванович",
    "fio_rod": "Иванова Ивана Ивановича",
    "position": "Заместитель начальника управления"
  }
]
```

Ключи произвольные — те, которые нужны для подстановки.

**Шаг Б.** В YAML-конфиге шаблона:
```yaml
- name: zam
  label: "Заместитель"
  type: dictionary
  source: deputies                  # имя файла без .json
  display: "{fio} ({position})"     # как показывать в выпадающем списке
  fills:
    Фам_Имя_Отч_зама_им_пад: fio_im      # переменная_в_шаблоне: ключ_в_справочнике
    Фам_Имя_Отч_зама_род_пад: fio_rod
    должность_зама: position
```

В шаблоне эти переменные используются как обычно: `{должность_зама} {Фам_Имя_Отч_зама_им_пад}`. Само поле `zam` в шаблоне НЕ нужно — оно только для UI.

#### Как добавить новые поля в существующий справочник

Структура каждой записи — плоский объект «ключ → строка». Когда нужны дополнительные данные (например, ещё падежи, телефон, e-mail, номер кабинета), просто **добавляем новые ключи во все записи** JSON-файла:

```json
[
  {
    "fio": "Иванов И.И.",
    "fio_im": "Иванов Иван Иванович",
    "fio_rod": "Иванова Ивана Ивановича",
    "fio_dat": "Иванову Ивану Ивановичу",
    "fio_vin": "Иванова Ивана Ивановича",
    "fio_tvor": "Ивановым Иваном Ивановичем",
    "position": "Заместитель начальника управления",
    "phone": "+7 (495) 123-45-67",
    "office": "404"
  }
]
```

Имена ключей произвольные — выбирайте те, по которым потом удобно вспомнить значение. Привычные конвенции: `_im / _rod / _dat / _vin / _tvor / _pred` для падежей.

**Как использовать новые поля в шаблоне.** В YAML того шаблона, где они нужны, добавьте:

1. В `fills` — связку «переменная в шаблоне → ключ справочника»:
   ```yaml
   fills:
     Фам_Имя_Отч_зама_им_пад: fio_im
     Фам_Имя_Отч_зама_дат_пад: fio_dat   # ← новое
     телефон_зама: phone                  # ← новое
   ```
2. В `fields` — описание полей формы, в которые fills будет писать (тип обычно `text`, без `required`):
   ```yaml
   - name: Фам_Имя_Отч_зама_дат_пад
     label: "ФИО зама (дат.п.) — заполнится автоматически"
     type: text
   - name: телефон_зама
     label: "Телефон зама"
     type: text
   ```
3. В `.docx` — обычные плейсхолдеры: `{Фам_Имя_Отч_зама_дат_пад}`, `{телефон_зама}`.
4. По желанию можно расширить `display`, чтобы новое поле было видно в комбобоксе:
   ```yaml
   display: "{fio} — {position}, кабинет {office}"
   ```

**Что если поле есть не во всех записях.** Если у части записей нового ключа нет, в `.docx` подставится пустая строка (благодаря `nullGetter` в генераторе). Приложение не упадёт. Для аккуратности лучше всё-таки заполнить новый ключ во всех записях — хотя бы `""`.

**Несовместимости между шаблонами.** Один и тот же `dictionaries/deputies.json` используется всеми шаблонами, которые на него ссылаются (`source: deputies`). Добавление новых ключей старые шаблоны не ломает: они продолжат использовать только те ключи, которые упомянуты у них в `fills`/`display`. **Переименование или удаление существующих ключей** — наоборот, ломает все шаблоны, где эти ключи использовались; перед таким изменением `grep -r 'fio_im' templates/` по `fills` и `display`.

После изменений в `dictionaries/*.json` — обязательно пересчитать `manifest.json` (шаг 4.2), иначе у пользователей справочник не обновится.

### 2.5. Обязательные поля и валидация

- `required: true` — поле обязательно для заполнения.
- Для `number` — автоматическая валидация на корректный числовой формат.
- Для `date` — автоматическая валидация на корректную дату.
- Кастомных regex-валидаторов пока нет (можно добавить, если потребуется).

### 2.5а. Конвенция формата (базовое правило)

Эта конвенция жёсткая — нарушает её только тот, кто специально хочет сломать русскую юридическую локаль:

- **Даты в `.docx`** — всегда `дд.мм.гггг` (например, `15.02.2026`). Не `2026-02-15`, не `February 15, 2026`. Калькуляторы возвращают даты в этом формате (см. `formatRuDate` в `delivery-overdue.ts`); фиксированные даты, которые юрист пишет прямо в Word, тоже в `дд.мм.гггг`.
- **Денежные суммы в `.docx`** — десятичный разделитель **запятая**, разделитель тысяч — пробел: `1 234,56`. Не `1234.56`, не `1,234.56`. Калькуляторы с денежными выходами форматируют значение строкой (`toLocaleString("ru-RU", …)`) перед записью в `outputs`; docxtemplater подставит как есть.
- **В формах пользователь вводит** числа уже с запятой — `DecimalInput` принимает и `1,5`, и `1.5`, хранит как `number`. Если число дальше идёт в `.docx` через калькулятор, тот его правильно отформатирует. Если идёт напрямую (поле `type:number` подставляется в `.docx` без калькулятора) — на текущий момент попадёт как `1234.56`; решается обёртыванием через калькулятор-форматтер или будущим атрибутом `format: "currency"` (TODO).

### 2.6. Калькуляторы (`type: calculator`)

Когда в документ нужно подставить значение, которое **вычисляется** из других полей формы (неустойка, проценты по ст. 395 ГК, итоговая сумма с НДС и т. п.), используется поле типа `calculator`. Сам пользователь это поле не редактирует — он видит карточку с результатом, который пересчитывается на лету при изменении входов.

**Минимальный пример:**

```yaml
- name: сумма_контракта
  label: "Сумма контракта, руб."
  type: number
  required: true

- name: дней_просрочки
  label: "Дней просрочки"
  type: number
  required: true

- name: ставка_цб
  label: "Ставка ЦБ, %"
  type: number
  required: true

- name: неустойка_блок
  label: "Расчёт неустойки"
  type: calculator
  calculator: penalty_44fz_part6
  inputs:
    сумма_контракта: сумма_контракта     # вход_калькулятора: имя_поля_формы
    дней_просрочки: дней_просрочки
    ставка_цб: ставка_цб
  outputs:
    сумма_неустойки: сумма_неустойки      # выход_калькулятора: имя_переменной_для_docx
    расчёт_подробно: расчёт_неустойки_текст
```

В шаблоне `.docx`:
```
Сумма неустойки: {сумма_неустойки} руб.
Расчёт: {расчёт_подробно}.
```

**Как читать `inputs` / `outputs`:**

- `inputs` — слева имя ВХОДА калькулятора (определяется кодом калькулятора), справа имя поля формы, откуда брать значение. Если у вас в форме поле называется иначе, чем калькулятор ждёт на входе — поправьте маппинг.
- `outputs` — слева имя ВЫХОДА калькулятора (определяется кодом), справа имя переменной, под которой результат окажется в `.docx`. То есть `outputs.сумма_неустойки: сумма_неустойки` значит «выход калькулятора `сумма_неустойки` положить в переменную шаблона `{сумма_неустойки}`».

**Доступные калькуляторы.** Реестр живёт в [`src/lib/calculators/`](../src/lib/calculators/), там же лежат юнит-тесты с примерами.

| id | Что считает | Входы | Выходы |
|----|-------------|-------|--------|
| `penalty_44fz_part6` | Пеня по 44-ФЗ ч.6 ст.34: 1/300 ставки ЦБ за каждый день просрочки от суммы | `сумма_контракта`, `дней_просрочки`, `ставка_цб` (в %) | `сумма_неустойки` (число), `расчёт_подробно` (текст с разбором формулы) |

**Если форма ещё не заполнена** — карточка показывает «Заполните поля выше для расчёта», переменные `outputs` в шаблон уходят пустыми. На каждое изменение входа калькулятор пересчитывается через 200 мс.

**Если нужный калькулятор отсутствует.** Не вычислять формулу в YAML и не дублировать её в нескольких шаблонах. Завести новый файл в [`src/lib/calculators/<id>.ts`](../src/lib/calculators/), экспортнуть `CalculatorDef` (формат — см. `types.ts`), зарегистрировать в [`index.ts`](../src/lib/calculators/index.ts), покрыть юнит-тестом. **Универсальные** калькуляторы переиспользуются между шаблонами по `id`; **специфичные под один шаблон** оформляются точно так же, просто упоминаются ровно в одном YAML. Для системы это одно и то же.

### 2.6а. Автоперенос дат-дедлайнов на рабочий день

По ст. 193 ГК РФ, если последний день срока приходится на нерабочий день, срок считается наступившим в ближайший следующий рабочий. Калькуляторы дат (на сегодня — `delivery_overdue`) делают этот перенос автоматически: если дедлайн выпал на субботу, воскресенье или официальный праздник из производственного календаря, выход `дата_дедлайна` уже сдвинут, а `дней_просрочки` считаются от сдвинутой даты.

**Юристу в `.docx` ничего дополнительно делать не нужно.** Подставляйте `{дата_дедлайна}` как обычно — там уже будет корректная рабочая дата. Не пытайтесь дублировать эту логику в Word через условные блоки.

**Откуда берутся праздники.** Файл [`dictionaries/holidays.json`](../dictionaries/holidays.json) — плоский массив строк `YYYY-MM-DD`. Туда заносятся официальные нерабочие дни РФ (праздники + стандартные переносы по ст. 112 ТК РФ) на ближайшие годы. Файл доезжает до пилотов через тот же механизм автообновления, что и шаблоны (`manifest.json`), — править его можно отдельно от кода.

При добавлении нового календарного года в `holidays.json`: после правки запустить `python3 scripts/build_manifest.py`, закоммитить пару `dictionaries/holidays.json` + `manifest.json`.

**Если файла нет (или сеть его не подтянула).** Калькуляторы работают в безопасном режиме: переносят только субботу/воскресенье, праздники — нет. Это деградация, а не отказ — юрист всё равно может проверить и вручную поправить дату в готовом `.docx`.

### 2.7. Суммы прописью (`text_output`)

Если рядом с числовой суммой в документе нужна её запись прописью — без отдельного калькулятора. У числового поля проставляется атрибут `text_output: <имя_парной_переменной>`, и форма автоматически положит туда строку с правильными склонениями («рубль / рубля / рублей», «копейка / копейки / копеек», женский род у тысяч).

```yaml
- name: сумма_контракта
  label: "Сумма контракта, руб."
  type: number
  required: true
  text_output: сумма_контракта_прописью
```

В шаблоне `.docx`:
```
Сумма контракта: {сумма_контракта} ({сумма_контракта_прописью}) руб.
```

Подставится так:
```
Сумма контракта: 3451,00 (три тысячи четыреста пятьдесят один рубль 00 копеек) руб.
```

Под полем формы появляется превью «Прописью: …», чтобы юрист сразу видел, что получится в документе.

**Ограничения:** `text_output` работает только для `type: number`. Имя `text_output` не должно совпадать с именем какого-либо другого поля формы. Округление — до копеек, отрицательные числа дают префикс «минус» (но в претензиях такое появляется редко).

### 2.8. Шаблоны претензий и анкета

В режиме «Претензии» юрист не выбирает шаблон из списка, а проходит короткую анкету (`templates/pretenzii/_wizard.yaml`), которая по ответам выводит на единственный нужный шаблон. Поэтому каждый шаблон претензий **обязан** нести блок `tags:` — набор ответов, под которые он подходит.

Текущие оси анкеты (`axes` в `_wizard.yaml`):

| Ось | Значения |
|-----|----------|
| `закон` | `44`, `223` |
| `процедура` | `аукцион`, `ед_поставщик` |
| `тип_поставки` | `первая`, `частичные` |
| `количество_частичных` | `1`–`4` (только при `тип_поставки: частичные`) |

Пример блока в YAML шаблона:
```yaml
tags:
  закон: 44
  процедура: аукцион
  тип_поставки: первая
```

**Строгие правила (проверяются при старте, иначе приложение блокируется):**
- Все оси из `tags` объявлены в `axes` анкеты.
- Указаны ровно те оси, что видимы для этой ветви ответов. При `тип_поставки: первая` ось `количество_частичных` **не указывается**; при `частичные` — **указывается**.
- Значение тега есть среди вариантов соответствующего вопроса.
- Нет двух шаблонов с одинаковым полным набором тегов (на комбинацию ответов — один шаблон).

Если под комбинацию ответов шаблона нет — юрист увидит экран «Шаблон в разработке». Это нормально: добавится шаблон с нужными тегами — анкета сама начнёт на него выводить.

Чтобы **добавить новую ось** в анкету, нужно расширить `axes` и `questions` в `_wizard.yaml` **и** дописать новый тег во все существующие шаблоны претензий (иначе валидатор заблокирует старт). Multiselect и запоминание ответов между запусками — не поддерживаются сознательно.

Локальная проверка: `npm run tauri dev -- --mode=pretenzii` — пройти анкету и убедиться, что нужная комбинация ведёт в форму, а остальные — на «в разработке».

### 2.9. Несколько документов из одной формы (`extra_templates`)

Один проход формы может выдавать основной документ **плюс** произвольное число дополнительных (например, претензия + служебные записки). Все они заполняются из одного и того же набора переменных — форма общая, поля — супермножество переменных всех документов.

```yaml
template: 44_аукцион_первая.docx
extra_templates:
  - template: служебка_направление.docx
    output_filename: { pattern: "Служебка_направление_{date}.docx" }
  - template: служебка_контроль.docx
    output_filename: { pattern: "Служебка_контроль_{date}.docx" }
fields:
  - ...   # супермножество переменных всех документов
```

- Файлы `extra_templates[].template` лежат в **той же папке**, что и основной `.docx`; имена не дублируются; расширение `.docx`. Существование проверяется при загрузке (нет файла — шаблон не появится).
- Условные блоки `{#field}` работают в каждом документе независимо: лишние для конкретной служебки переменные просто не используются в её тексте.
- В форме кнопка отправки превращается в «Сформировать N документов». При N > 1 запрашивается **папка** (а не «Сохранить как…»), файлы пишутся туда по своим `output_filename.pattern`. Если хоть один документ не сформировался (битый плейсхолдер) — не пишется ни один, черновик формы сохраняется.

Кода генератора расширять не нужно — связку собирает `src/lib/generator.ts` (`generateBundle`).

## Шаг 3. Валидация

### 3.1. Извлечение переменных из шаблона

```bash
python scripts/extract_variables.py путь/к/шаблону.docx
```

Скрипт:
- Парсит `.docx`, находит все плейсхолдеры в тексте, таблицах, колонтитулах.
- Печатает список найденных переменных.
- **Предупреждает о разорванных плейсхолдерах** (см. 1.4) — это первая проверка корректности файла.
- Опционально генерирует заготовку YAML-конфига.

Если скрипт ругается на разорванные плейсхолдеры — открыть `.docx` в Word, найти проблемные места, применить `Ctrl+Space` или ввести плейсхолдер заново.

### 3.2. Сверка с YAML

```bash
python scripts/validate_template.py templates/{mode}/{name}.docx templates/{mode}/{name}.yaml
```

Скрипт сверяет:
- Все переменные из `.docx` либо описаны в `fields` YAML, либо приходят через `fills` из dictionary-поля.
- Все поля из YAML (которые не помечены как чисто UI вроде `dictionary` без подстановки) реально используются в `.docx`.
- `output_filename.fields` — все поля существуют и помечены `required: true`.
- `visible_if` ссылается на существующие поля.

Все расхождения — печатает с указанием места.

### 3.3. Локальная проверка в приложении

1. Скопировать `.docx` + `.yaml` во временную папку `templates/{mode}/` в локальной сборке.
2. Запустить `npm run tauri dev -- --mode={mode}`.
3. Открыть новый шаблон, заполнить тестовыми данными, сгенерировать `.docx`.
4. Открыть результат в Word, проверить визуально:
   - Все переменные подставились (нет оставшихся `{переменная}` в тексте).
   - Условные блоки сработали корректно (заполнить форму один раз с «Да», другой раз с «Нет», сравнить).
   - Форматирование сохранено (шрифты, таблицы, отступы).
5. Поправить, если что-то не так — и снова прогнать.

## Шаг 4. Публикация

После того как шаблон работает локально:

### 4.1. Положить файлы в репозиторий

```
templates/
├── pretenzii/
│   └── pretenziya_po_srokam.docx   ← сюда
│   └── pretenziya_po_srokam.yaml   ← и сюда
```

Если в шаблоне используются новые ключи справочника — обновить соответствующий `dictionaries/*.json`.

### 4.2. Пересчитать манифест

```bash
python scripts/build_manifest.py
```

Скрипт обходит `templates/` и `dictionaries/`, считает SHA-256 для каждого файла, обновляет `manifest.json` в корне репозитория.

### 4.3. Закоммитить и запушить

```bash
git add templates/ dictionaries/ manifest.json
git commit -m "Добавлен шаблон: {название}"
git push
```

После пуша — при следующем запуске приложения у любого сотрудника (с интернетом) шаблон появится автоматически. Перевыпуск приложения не нужен.

## Шаг 5. Изменения в существующих шаблонах

Если меняется текст существующего шаблона (мелкая правка, опечатки, новая редакция):

1. Открыть `.docx`, отредактировать в Word.
2. Если правки затрагивают плейсхолдеры — прогнать `extract_variables.py` и `validate_template.py`.
3. Если меняются поля формы — обновить `.yaml`.
4. Локально проверить.
5. `python scripts/build_manifest.py` — обязательно, иначе пользователи не увидят обновление.
6. `git commit`, `git push`.

**Важно про черновики у пользователей.** Если меняется набор полей формы (добавили/убрали/переименовали), у пользователей могут оставаться черновики со старыми полями. При восстановлении приложение их сверит и покажет предупреждение «часть полей не подставится» — это ожидаемое поведение.
