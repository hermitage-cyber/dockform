# Выпуск релиза

Эта инструкция описывает процесс выпуска новой версии приложения: от bump версии до проверки, что обновление дошло до пользователей. Выполняется редко (по мере накопления изменений), поэтому идём строго по шагам и не импровизируем — ошибка в подписи или манифесте обновления означает, что у пользователей перестанет работать автообновление.

## Когда выпускать релиз

- Закончен этап разработки из `plan.md` (например, Этап 5 → выпуск v0.5.0).
- Накопились правки кода (НЕ шаблонов — для шаблонов релиз приложения не нужен, см. `template-authoring.md`).
- Срочная правка критичного бага.

**НЕ выпускать релиз только ради:**
- Изменения шаблонов или справочников — они доставляются через `manifest.json` отдельно.
- Изменения `killswitch.json` — он читается напрямую с GitHub.

## Версионирование

Семантическое: `MAJOR.MINOR.PATCH`.
- **MAJOR** — несовместимые изменения формата шаблонов или конфигурации (потребует ручных действий от пользователей).
- **MINOR** — новые возможности, совместимые со старыми шаблонами.
- **PATCH** — багфиксы.

До v1.0.0 — все этапы из `plan.md` идут как `0.N.0` (Этап 1 = v0.1.0, Этап 2 = v0.2.0, …). После Этапа 7 и пилотной проверки — v1.0.0.

## Подготовка

### Шаг 1. Проверить, что всё работает

1. `npm run tauri dev` — приложение запускается без ошибок в консоли.
2. Прогнать чеклист текущего этапа из `plan.md` (что должно работать после этого этапа).
3. Прогнать «дымовой тест»: запустить, выбрать шаблон, заполнить, сохранить — документ создаётся корректно.
4. Если в этом релизе менялись сетевые функции — проверить с реально включенным и выключенным интернетом.

Если что-то не работает — НЕ выпускать релиз, доделать.

### Шаг 2. Bump версии

Версия указана в трёх местах, они должны быть синхронны:

```
package.json              → "version": "0.5.0"
src-tauri/Cargo.toml      → version = "0.5.0"     (под [package])
src-tauri/tauri.conf.json → "version": "0.5.0"
```

Поправить руками во всех трёх. После правки:
```bash
git add package.json src-tauri/Cargo.toml src-tauri/tauri.conf.json
git commit -m "Bump версии до 0.5.0"
```

### Шаг 3. Обновить плейсхолдер «Текущий этап» в CLAUDE.md

Раз с релизом меняется этап разработки — поправить секцию «Текущий этап» в `CLAUDE.md`, чтобы Claude Code в следующих сессиях знал, на чём сосредоточиться.

```bash
git add CLAUDE.md
git commit -m "CLAUDE.md: обновлён текущий этап"
```

### Шаг 4. Запушить изменения

```bash
git push
```

Версия и контекст в main, готовы к сборке.

## Сборка и подпись

### Шаг 5. Получить ключ для подписи

Приватный ключ для подписи релизов хранится **не в репозитории**. Где он лежит у вас — зависит от вашей системы (менеджер паролей, зашифрованный файл, отдельный USB). Проверьте, что он доступен.

Если ключа ещё нет (самый первый релиз) — сгенерировать:
```bash
npm run tauri signer generate -- -w ~/.tauri/dockform.key
```

Скрипт спросит парольную фразу — придумать и запомнить (или сохранить в менеджер паролей).

Команда напечатает:
- Приватный ключ → файл `~/.tauri/dockform.key` (положить в надёжное место + зашифрованный бэкап).
- Публичный ключ → строка, вставить её в `src-tauri/tauri.conf.json`:
  ```json
  {
    "plugins": {
      "updater": {
        "pubkey": "ВСТАВИТЬ_СЮДА",
        "endpoints": ["https://github.com/{owner}/{repo}/releases/latest/download/latest.json"]
      }
    }
  }
  ```
- Закоммитить tauri.conf.json с публичным ключом.

**⚠️ Если приватный ключ потерян** — выпустить новую версию автообновлением не получится. Понадобится сгенерировать новую пару, обновить публичный ключ в коде, и доставить эту версию пользователям вручную (через флешку, общий ресурс). Дальше автообновление снова заработает.

### Шаг 6. Установить переменные окружения для подписи

В терминале, где будете запускать сборку:

```bash
# Windows PowerShell
$env:TAURI_SIGNING_PRIVATE_KEY = Get-Content C:\path\to\dockform.key -Raw
$env:TAURI_SIGNING_PRIVATE_KEY_PASSWORD = "ваша_парольная_фраза"

# Windows CMD
set TAURI_SIGNING_PRIVATE_KEY=...
set TAURI_SIGNING_PRIVATE_KEY_PASSWORD=...
```

Эти переменные нужны только в окне сборки и не должны попасть в git, в логи, в историю команд (по возможности).

### Шаг 7. Запустить сборку

```bash
npm run tauri build
```

Сборка займёт 3–10 минут (Rust компилируется небыстро при первом разе и при больших изменениях).

После успешной сборки артефакты лежат в:
```
src-tauri/target/release/bundle/nsis/
├── dockform_0.5.0_x64-setup.nsis.zip
├── dockform_0.5.0_x64-setup.nsis.zip.sig
└── dockform_0.5.0_x64-setup.exe          ← инсталлятор
```

Расширения могут отличаться в зависимости от настройки `bundle.targets` в `tauri.conf.json`. По умолчанию для Windows — NSIS.

Также рядом будет автоматически сгенерированный `latest.json`:
```json
{
  "version": "0.5.0",
  "notes": "См. CHANGELOG",
  "pub_date": "2026-05-15T10:00:00Z",
  "platforms": {
    "windows-x86_64": {
      "signature": "длинная_строка_подписи",
      "url": "https://github.com/{owner}/{repo}/releases/download/v0.5.0/dockform_0.5.0_x64-setup.nsis.zip"
    }
  }
}
```

⚠️ В этом файле URL пока **неправильный** — он будет таким после загрузки артефакта на GitHub. На этом шаге это просто шаблон.

### Шаг 8. Проверить локально, что сборка работает

1. Запустить собранный `dockform.exe` (он лежит в `src-tauri/target/release/`).
2. Проверить базовый сценарий: выбор шаблона → заполнение → генерация.
3. Если работает — продолжаем. Если нет — расследовать, не выпускать.

## Публикация на GitHub Releases

### Шаг 9. Создать тег

```bash
git tag v0.5.0
git push origin v0.5.0
```

Префикс `v` — обязательно, иначе путаница с папками и URL.

### Шаг 10. Создать release на GitHub

Через UI GitHub:
1. Repo → Releases → «Draft a new release».
2. Tag → выбрать `v0.5.0`.
3. Release title: `v0.5.0 — краткое описание этапа`.
4. Description: список изменений в этом релизе (что сделали, что починили).
5. Загрузить файлы из `src-tauri/target/release/bundle/nsis/`:
   - `dockform_0.5.0_x64-setup.nsis.zip`
   - `dockform_0.5.0_x64-setup.nsis.zip.sig`
   - `latest.json` (откорректировать URL внутри, если нужно — заменить на финальный GitHub release URL).
6. **НЕ ставить галку «Pre-release»** — `tauri-plugin-updater` смотрит на `latest`, который указывает на последний non-prerelease релиз.
7. «Publish release».

Альтернатива через `gh` CLI:
```bash
gh release create v0.5.0 \
  src-tauri/target/release/bundle/nsis/dockform_0.5.0_x64-setup.nsis.zip \
  src-tauri/target/release/bundle/nsis/dockform_0.5.0_x64-setup.nsis.zip.sig \
  src-tauri/target/release/bundle/nsis/latest.json \
  --title "v0.5.0 — описание" \
  --notes "Список изменений"
```

### Шаг 11. Проверить, что latest.json доступен

Открыть в браузере:
```
https://github.com/{owner}/{repo}/releases/latest/download/latest.json
```

Файл должен загрузиться. URL внутри файла должен указывать на тот же релиз (`/download/v0.5.0/dockform_...`).

## Проверка обновления

### Шаг 12. Прогнать обновление у тестового пользователя

1. Взять чистую установку с прошлой версией (например, v0.4.0) — на отдельной машине или в виртуалке. Можно использовать вашу же машину, если предварительно положить туда старую версию в отдельную папку.
2. Запустить — должен появиться тост «Доступна версия 0.5.0. Установить?».
3. Согласиться — приложение скачивает обновление, проверяет подпись, устанавливает, перезапускается.
4. Проверить, что в открывшемся приложении версия 0.5.0 (например, в about-диалоге, который нужно будет добавить как фичу, либо в заголовке окна на этапе разработки).

Если обновление не приходит — диагностика:
- `latest.json` действительно отдаётся по URL (см. шаг 11)?
- Подпись валидна (`.sig` файл лежит рядом с архивом и доступен)?
- Версия в `latest.json` больше текущей у клиента?

Если обновление не устанавливается:
- Подпись не сходится (выпускали с другим приватным ключом)?
- Архив не скачивается (битый URL)?
- В логах Tauri (`%LOCALAPPDATA%\com.dockform.app\logs\` или путь, заданный приложением) есть подробности.

## Откат

Если выпустили битый релиз и автообновление уже доставило его пользователям:

1. **НЕ удалять** битый релиз с GitHub — это сломает уже скачанные `latest.json`-кеши у клиентов.
2. Срочно собрать фикс-версию (bump до следующего patch — например, 0.5.1).
3. Выпустить её как обычный релиз. Пользователи обновятся при следующем запуске.

В качестве временной меры можно активировать kill switch (`killswitch.json` → `active: false`), чтобы остановить распространение проблемы, пока готовится фикс.
