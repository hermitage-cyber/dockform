# Выпуск релиза

Инструкция, как собрать новую версию портативного `dockform.exe` и раздать её пользователям. Релиз делается редко (по мере накопления правок кода), идём строго по шагам.

Это **portable**-workflow: устанавливать ничего не нужно, файл запускается с флешки или из любой папки. Автообновление приложения через `tauri-plugin-updater` в текущем дистрибутиве **отключено** (см. ниже «Почему без автообновлений»), поэтому ключи подписи и `latest.json` не используются.

## Когда выпускать релиз

- Закончен этап разработки из `plan.md` (например, Этап 8 → выпуск v0.8.0).
- Накопились правки кода (НЕ шаблонов — для шаблонов релиз приложения не нужен, см. `template-authoring.md`).
- Срочная правка критичного бага.

**НЕ выпускать релиз только ради:**
- Изменения шаблонов или справочников — они доставляются через `manifest.json` отдельно, без пересборки приложения.
- Изменения `killswitch.json` — он читается напрямую с GitHub.

## Версионирование

Семантическое: `MAJOR.MINOR.PATCH`.
- **MAJOR** — несовместимые изменения формата шаблонов или конфигурации (потребует ручных действий от пользователей).
- **MINOR** — новые возможности, совместимые со старыми шаблонами.
- **PATCH** — багфиксы.

До v1.0.0 — этапы `plan.md` идут как `0.N.0` (Этап 8 = v0.8.0, Этап 9 = v0.9.0). После прохождения пилота — v1.0.0.

## Подготовка

### Шаг 1. Проверить, что всё работает

1. `npm run tauri dev` — приложение запускается без ошибок в консоли.
2. Прогнать чеклист текущего этапа из `plan.md`.
3. Дымовой тест: выбрать шаблон (или пройти анкету в режиме претензий), заполнить, сохранить — документ создаётся корректно и открывается в Word.
4. Если в этом релизе менялись сетевые функции (kill switch, обновление шаблонов) — проверить с включённым и выключенным интернетом.

Если что-то не работает — НЕ выпускать релиз, доделать.

### Шаг 2. Bump версии

Версия указана в трёх местах, они должны быть синхронны:

```
package.json              → "version": "0.9.0"
src-tauri/Cargo.toml      → version = "0.9.0"     (под [package])
src-tauri/tauri.conf.json → "version": "0.9.0"
```

Поправить руками во всех трёх. После правки:
```bash
git add package.json src-tauri/Cargo.toml src-tauri/tauri.conf.json
git commit -m "Bump версии до 0.9.0"
```

### Шаг 3. Обновить «Текущий этап» в CLAUDE.md

Раз с релизом меняется этап разработки — поправить секцию «Текущий этап» в `CLAUDE.md`, чтобы Claude Code в следующих сессиях знал, на чём сосредоточиться.

```bash
git add CLAUDE.md
git commit -m "CLAUDE.md: обновлён текущий этап"
git push
```

## Сборка

### Шаг 4. Собрать на Windows-машине

Сборка под Windows — на Windows. Кросс-компиляция с macOS/Linux не поддерживается официальным toolchain'ом Tauri и в этом проекте не используется.

На рабочей Windows-машине с установленными Rust, Node.js, npm, Visual Studio Build Tools:

```bash
git pull
npm install
npm run tauri build
```

Сборка займёт 3–10 минут (Rust компилируется небыстро при первом разе и при больших изменениях).

После успешной сборки портативный бинарь лежит здесь:

```
src-tauri/target/release/dockform.exe
```

Это и есть наш portable. В папке `src-tauri/target/release/bundle/` Tauri дополнительно положит NSIS-инсталлятор и MSI — мы их **не используем** для распространения, можно игнорировать или удалить.

### Шаг 5. Локальная проверка сборки

1. Скопировать `dockform.exe` в отдельную пустую папку вне репозитория.
2. Рядом — папки `templates/` и `dictionaries/` (просто скопировать соответствующие каталоги из репо).
3. Запустить `dockform.exe`. Должно открыться окно. Проверить базовый сценарий: выбор шаблона / анкета → заполнение → генерация документа.
4. Если работает — продолжаем. Если нет — расследовать, не выпускать.

## Подготовка папки распространения

### Шаг 6. Собрать финальную папку

Структура того, что отдаём пользователям:

```
dockform/
├── dockform.exe
├── Претензионная.lnk         (создаётся скриптом, шаг 7)
├── Документация.lnk          (создаётся скриптом, шаг 7)
├── README.txt                (копия docs/portable-README.txt)
├── templates/
│   ├── pretenzii/
│   └── documentation/
├── dictionaries/
├── cache/                    (пустая, для черновиков формы и сетевых кэшей)
└── output/                   (пустая, для готовых .docx)
```

Шаги:
1. Создать пустую папку `dockform/` где удобно.
2. Скопировать туда `dockform.exe` из `src-tauri/target/release/`.
3. Скопировать `templates/` и `dictionaries/` из репозитория.
4. Создать пустые папки `cache/` и `output/`.
5. Скопировать `docs/portable-README.txt` в корень папки под именем `README.txt`.

### Шаг 7. Сгенерировать ярлыки

В папке `dockform/` (где лежит `dockform.exe`) запустить:

```powershell
powershell -ExecutionPolicy Bypass -File <путь_к_репо>\scripts\make_shortcuts.ps1
```

Скрипт создаст два `.lnk`-файла: «Претензионная.lnk» (`--mode=pretenzii`) и «Документация.lnk» (`--mode=documentation`).

Если запускать скрипт не из папки портабла — передать путь явно:
```powershell
powershell -ExecutionPolicy Bypass -File scripts\make_shortcuts.ps1 -PortableDir D:\dist\dockform
```

### Шаг 8. Упаковать в zip

```powershell
Compress-Archive -Path dockform -DestinationPath dockform_0.9.0.zip
```

Внутри архива должна оказаться одна папка `dockform/` со всем содержимым. Проверить:
```powershell
Expand-Archive -Path dockform_0.9.0.zip -DestinationPath проверка\
dir проверка\dockform
```

## Раздача

### Шаг 9. Доставить пользователям

Способы (выбрать удобный):
- Положить `dockform_0.9.0.zip` на общий сетевой ресурс / Google Drive / Dropbox, разослать ссылку.
- Записать на флешки, раздать лично.
- Отправить почтой (если размер позволяет).

Сопроводить коротким сообщением:
- Распаковать архив в любую папку (Документы, флешка, рабочий стол — без разницы).
- Запускать через ярлыки «Претензионная» / «Документация».
- Если ярлык не открывается — см. `README.txt` в папке.

### Шаг 10. Создать тег в репозитории

Для истории и удобства поиска версии:

```bash
git tag v0.9.0
git push origin v0.9.0
```

Префикс `v` — обязательно.

Опционально создать GitHub Release с тегом и приложенным zip-архивом — это удобно как зеркало для пользователей, у которых пропала локальная копия. Через `gh`:
```bash
gh release create v0.9.0 dockform_0.9.0.zip \
  --title "v0.9.0 — Этап 9: упаковка portable" \
  --notes "Список изменений..."
```

## Почему без автообновлений

В реализации этапа 6.3 (`plan.md`) подключён `tauri-plugin-updater`, но в текущем дистрибутиве он **не используется**:
- В `tauri.conf.json` стоит `"createUpdaterArtifacts": false` — `.sig` и `latest.json` не создаются.
- `pubkey` в `plugins.updater` оставлен заглушкой (`REPLACE_BEFORE_FIRST_RELEASE`) — реальный ключ не сгенерирован.
- При запуске фронт вызывает `check()` (см. `src/App.tsx`), но без валидной пары ключ/`latest.json` вызов завершается ошибкой, которая ловится в `try/catch` и просто пишется в `console.warn` — UI не блокируется.

Причины такого решения для пилотной версии:
- Целевая аудитория — 5–10 сотрудников, новую версию проще доставить лично или общим ресурсом.
- Portable-сценарий (exe рядом с папкой `templates/` на флешке) плохо сочетается со стандартной моделью `tauri-plugin-updater`, рассчитанной на установленное приложение в `Program Files`.
- Удалённая блокировка (`killswitch.json`) и обновление шаблонов остаются — критичные защитные механизмы работают.

Если в будущем понадобится автообновление приложения — порядок включения:
1. Сгенерировать пару ключей: `npm run tauri signer generate -- -w ~/.tauri/dockform.key`.
2. Вставить публичный ключ в `src-tauri/tauri.conf.json` → `plugins.updater.pubkey`.
3. Поменять `bundle.createUpdaterArtifacts` на `true`.
4. Переключить распространение с portable zip на NSIS-инсталлятор (`bundle/nsis/*.exe`), либо разобраться, как updater работает поверх portable-папки (нетривиально).
5. Каждый релиз — выгружать на GitHub Releases `latest.json`, `.nsis.zip` и `.sig`.

Подробности — этап 6.3 в `plan.md`.

## Откат

Если выпустили битый релиз и архив уже у пользователей:

1. Сообщить пилотам, что текущая версия проблемная (мессенджер / телефон), попросить временно не пользоваться.
2. На случай удалённой блокировки можно поднять флаг в репозитории:
   ```json
   // killswitch.json
   { "active": false, "message": "Версия 0.9.0 содержит ошибку, ждите обновление." }
   ```
   Закоммитить и запушить. При следующем запуске у пользователей экран блокируется.
3. Быстро собрать фикс-версию (bump до следующего patch — например, 0.9.1), повторить релиз.
4. Снять флаг блокировки и раздать новый архив.
